Orbit 实验中心下一步清单与新增能力设计
本文档面向后续研发实现和 AI 编码工具理解，基于提交 944dc803 feat: add experiment center workflow 后的代码状态编写。
1. 当前状态
   实验中心第一阶段已经具备以下能力：
   后端模型：Experiment、ExperimentRun、RunMetric、RunArtifact。
   后端 API：实验 CRUD、Run 查询、Metric/Param/Tag/Artifact 上报、Run finish。
   Job 接入：PyTorch、TensorFlow、Custom、Jupyter、WebIDE 创建时可绑定实验并创建 Run。
   环境变量注入：ORBIT_RUN_ID、ORBIT_RUN_TOKEN、ORBIT_API_BASE。
   Python SDK：轻量 telemetry SDK，支持参数、指标、产物、finish、offline jsonl。
   前端：实验列表、实验详情、Run 详情、曲线对比。
   当前第一阶段更准确地说是 Experiment Telemetry v1，还不是完整的实验复现中心。下一步重点是把 checkpoint、数据版本、权限、恢复链路和 SDK 发布补齐。
2. 现有代码需要修改的清单
   以下项目都是基于现有功能的修补或增强，只说明改哪里、改什么、怎么改。
   2.1 修复 checkpoint 恢复任务复用旧 Run 的问题
   问题：
   RestoreJobFromRecord 会复制历史 Job 模板。
   历史模板里的容器 env 可能包含旧的 ORBIT_RUN_ID、ORBIT_RUN_TOKEN、ORBIT_API_BASE。
   checkpoint 恢复任务目前只重写 checkpoint env，没有创建新 experiment run，也没有清理旧 run env。
   结果可能是恢复出来的新 Job 继续把指标上报到旧 run。
   改哪里：
   backend/internal/handler/vcjob/checkpoint_runtime.go
   backend/internal/service/vcjob/runtime.go
   backend/internal/handler/vcjob/util.go
   怎么改：
   在 checkpoint restore 流程里创建新的 experiment run。
   从源 Job annotation 读取 orbit.raids.io/experiment-id。
   如果源 Job 没有关联 experiment，则恢复任务不创建 experiment run，但必须清理旧 ORBIT_RUN_* env。
   如果有关联 experiment，则调用 ExperimentService.CreateRun。
   新 run 的 RunName 默认使用恢复任务展示名，例如 <source-run-name>-resume 或 <job-name>-resume。
   Tags 增加 restoredFromRunID、restoredFromJobName、restoredFromCheckpointID。
   CheckpointSnapshot 写入 checkpoint path、checkpoint ID、step、latest。
   恢复任务容器 env 处理顺序调整为：
   先清理历史 ORBIT_RUN_* 和 ORBIT_API_BASE。
   再注入新的 experiment env。
   再注入 checkpoint env，或者保持 checkpoint env 最后覆盖同名 checkpoint 变量。
   给 buildCheckpointRestoreJob 增加参数：
   experimentRuntime *experimentRunRuntime
   或者在函数内部创建并返回 runtime。
   为恢复 Job 的 annotation 设置新的 orbit.raids.io/experiment-run-id。
   验收：
   从有实验绑定的 Job checkpoint 恢复后，会生成新的 ExperimentRun。
   新恢复 Job 的容器 env 里只有新的 ORBIT_RUN_ID 和 ORBIT_RUN_TOKEN。
   新恢复 Job 的指标不会写到源 run。
   从无实验绑定 Job 恢复时，恢复 Job 不包含任何旧 ORBIT_RUN_*。
   2.2 修复账户可见实验下 Run 权限不一致
   问题：
   ListRuns 按 experiment visibility 放行。
   GetRun、ListMetrics、ListArtifacts 只允许 run owner 或平台管理员。
   账户可见实验中，其他账户成员能看到 run 列表，但点详情可能 404。
   改哪里：
   backend/internal/service/experiment_service.go
   backend/internal/service/experiment_service_test.go
   怎么改：
   新增方法 canAccessRun(ctx, runID, token)。
   查询 run 时 preload 或 join experiment。
   访问规则改成：
   同 account。
   run owner 可以访问。
   平台管理员可以访问。
   如果 run 所属 experiment visibility 是 account，同账户成员可以访问。
   GetRun、ListMetrics、ListArtifacts 都使用同一访问规则。
   增加单测覆盖：
   私有实验：非 owner 不可访问 run。
   账户实验：同 account 用户可访问 run/metrics/artifacts。
   跨 account 不可访问。
   验收：
   账户可见实验的 run 列表、详情、指标、产物权限表现一致。
   2.3 修复 PyTorch/TensorFlow 分布式 Job 的 imageSnapshot 为空
   问题：
   prepareExperimentRun 通过 toJSONMap(imageSnapshot) 保存镜像快照。
   PyTorch/TensorFlow 传入的是 req.Tasks 数组。
   数组无法 unmarshal 到 datatypes.JSONMap，最终得到 {}。
   改哪里：
   backend/internal/handler/vcjob/util.go
   backend/internal/handler/vcjob/pytorch.go
   backend/internal/handler/vcjob/tensorflow.go
   怎么改：
   新增 toJSONObject(value any) datatypes.JSONMap，支持数组包装：
   map/object：原样返回。
   slice/array：返回 {"items": [...]}。
   scalar：返回 {"value": ...}。
   或者在 PyTorch/TensorFlow 调用处明确构造：
   ImageSnapshot: {"tasks": req.Tasks}。
   推荐第二种，语义更清楚。
   验收：
   PyTorch/TensorFlow Run 详情中 imageSnapshot.tasks 能看到各 task image。
   2.4 给数据版本补 UI 到 payload
   问题：
   后端 ExperimentRunConfig 和 ExperimentRun 已有 DataSnapshot。

   前端表单展示“数据快照 JSON”字段。
   验收：
   创建 Job 绑定实验时，可以通过 UI 写入数据快照。
   Run 详情页展示 dataSnapshot。
   2.5 在 Run 详情页展示 dataSnapshot 和 tags

   2.7 SDK sync 避免重复写入
   问题：
   orbit.sync() 重放 offline jsonl 时，后端没有幂等键。
   重复 sync 会重复插入 metrics/artifacts。
   改哪里：
   sdk/python/orbit/client.py
   backend/dao/model/experiment.go
   backend/internal/service/experiment_service.go
   backend/cmd/gorm-gen/models/migrate.go
   怎么改：

   2.8 实验服务测试隔离配置初始化

   怎么改：
   优先让实验服务测试不触发全局 config init。
   如果无法避免，在测试中设置 ORBIT_DEBUG_CONFIG_PATH=./etc/example-config.yaml。
   更好的长期方案是拆分 service 包中会触发 config init 的依赖。
   验收：
   新增实验服务单测可以在 clean repo 中独立运行。
3. 新增能力设计一：Experiment 与 Checkpoint 打通
   3.1 背景
   当前 checkpoint 子系统和实验中心是两套并行能力：
   checkpoint 子系统知道 Job 下有哪些 checkpoint 文件。
   实验中心知道 Run 的指标、参数、产物。
   但 checkpoint 没有直接归属到 Run。
   用户在实验中心里仍然难以回答：
   这个 Run 有哪些 checkpoint？
   最好指标对应哪个 checkpoint？
   从这个 checkpoint 恢复会产生哪个新 Run？
   这个 Run 是否可以复现？
   3.2 目标
   每个 checkpoint 可以关联到一个 ExperimentRun。
   checkpoint scanner 扫到 checkpoint 后，自动把 checkpoint 注册成 Run artifact。
   Run 详情页显示 checkpoint 列表。
   checkpoint 恢复后创建新的 Run，并保留来源关系。
   不要求第一版上传/托管 checkpoint 文件，只记录路径和元数据。
   3.3 非目标
   不实现 torch.save / torch.load。
   不实现 checkpoint 文件上传。
   不实现异步 checkpoint。
   不实现模型注册中心。
   3.4 数据模型
   修改 backend/dao/model/job_checkpoint.go：

type JobCheckpoint struct {
// existing fields
RunID *uint `json:"runID" gorm:"index;comment:关联的实验 Run ID"`
}
修改 backend/dao/model/experiment.go：

type RunArtifact struct {
// existing fields
SourceType string `json:"sourceType" gorm:"type:varchar(64);index;comment:来源类型，如 checkpoint/manual/sdk"`
SourceID   *uint  `json:"sourceID" gorm:"index;comment:来源记录 ID，例如 JobCheckpoint.ID"`
}
可选新增 Run lineage 字段：

type ExperimentRun struct {
ParentRunID      *uint `json:"parentRunID" gorm:"index"`
SourceCheckpointID *uint `json:"sourceCheckpointID" gorm:"index"`
}
如果希望保持第一版小改动，也可以先把来源关系写入 Tags：

{
"parentRunID": 123,
"sourceCheckpointID": 456,
"sourceCheckpointPath": "/workspace/checkpoints/checkpoint-1000.pt"
}

3.5.1 创建 Job 时

3.5.2 scanner 扫描 checkpoint 时
改动位置：

实现逻辑：
从 Job record 或 Job annotation 读取 experiment run ID。
创建 JobCheckpoint 时设置 RunID。

artifact 规则：
name: checkpoint name
type: checkpoint
path: checkpoint path
sizeBytes: checkpoint size

metadata: framework、step、latest、storagePath、modTime、jobName
如果 checkpoint 已经注册过 artifact，不重复创建。

4. 新增能力设计二：Checkpoint SDK
   4.1 背景
   当前平台只负责 checkpoint 路径注入、扫描和恢复任务创建，不负责真正保存/加载模型状态。用户仍然需要自己写：

torch.save(...)
torch.load(...)
model.load_state_dict(...)
这导致：
用户接入成本高。
checkpoint 格式不统一。
平台无法判断 checkpoint 是否完整。
异步 checkpoint 无法统一支持。
4.2 目标
提供 Python SDK 的 checkpoint 模块。
第一版支持 PyTorch。
保持 SDK core 框架无关。
保存 checkpoint 时生成 manifest 和 latest marker。
支持同步保存和可选异步保存。
与实验中心 artifact 打通。
4.3 非目标
第一版不支持 TensorFlow/JAX/DeepSpeed/FSDP 分片保存。
第一版不做对象存储上传。
第一版不替代现有 scanner。
4.4 SDK API
新增模块：

sdk/python/orbit/checkpoint.py
sdk/python/orbit/pytorch.py
通用 API：

import orbit

ctx = orbit.init()
orbit.checkpoint.record(
path="/workspace/checkpoints/checkpoint-1000.pt",
step=1000,
metadata={"framework": "custom"}
)
orbit.checkpoint.flush()
PyTorch API：

import orbit
import orbit.pytorch as orbit_torch

orbit.init()

state = orbit_torch.load_checkpoint_if_available(
model=model,
optimizer=optimizer,
scheduler=scheduler,
scaler=scaler,
map_location="cpu",
)

start_step = state.step if state else 0

orbit_torch.save_checkpoint(
model=model,
optimizer=optimizer,
scheduler=scheduler,
scaler=scaler,
step=step,
epoch=epoch,
hparams=hparams,
async_save=True,
)

orbit.checkpoint.flush()
orbit.finish("succeeded")

4.6 Checkpoint 文件格式
PyTorch checkpoint 内容：

{
"schema_version": "orbit.pytorch.checkpoint.v1",
"model": model.state_dict(),
"optimizer": optimizer.state_dict() if optimizer else None,
"scheduler": scheduler.state_dict() if scheduler else None,
"scaler": scaler.state_dict() if scaler else None,
"step": step,
"epoch": epoch,
"hparams": hparams or {},
"metadata": metadata or {},
}
manifest 文件：

{
"schemaVersion": "orbit.checkpoint.manifest.v1",
"framework": "pytorch",
"name": "checkpoint-1000.pt",
"path": "/workspace/checkpoints/checkpoint-1000.pt",
"step": 1000,
"createdAt": "2026-06-02T12:00:00Z",
"sizeBytes": 123456,
"sha256": "optional",
"runID": 123,
"jobName": "pyt-user-xxx"
}
4.7 原子写入
保存流程：
写入临时文件：checkpoint-1000.pt.tmp。
写入完成后 rename 为 checkpoint-1000.pt。
写入 manifest：checkpoint-1000.pt.orbit.json。
更新 latest_checkpointed_iteration.txt。
调用 orbit.log_artifact(..., type="checkpoint")。
scanner 应跳过：
以 . 开头。
以 _tmp 开头。
以 .tmp 结尾。
4.8 异步保存
设计：
async_save=False：当前线程直接 torch.save。
async_save=True：主线程先构造 CPU snapshot，后台线程写文件。
flush() 等待所有后台保存完成。
限制：
第一版只保证单进程或 DDP rank 0 保存。
如果用户传入巨大模型，CPU snapshot 可能占用大量内存，需要文档提示。
4.9 DDP 支持
SDK 规则：
默认只在 rank 0 保存。
rank 判断来源：
RANK
LOCAL_RANK
WORLD_SIZE
非 rank 0 调用 save 时直接 no-op。
可选参数 save_on_all_ranks=False。
4.10 验收标准
PyTorch 单进程训练可保存和恢复。
DDP 场景只有 rank 0 写 checkpoint。
保存完成后 scanner 能扫到 checkpoint。
Run 详情页能看到 checkpoint artifact。
async_save=True 时训练可以继续执行，flush() 能等待落盘。
5. 新增能力设计三：Run 复现入口
   5.1 背景
   实验中心的最终价值是让用户能从历史 run 回答：
   哪次效果最好？
   这次 run 为什么好？
   能不能复现？
   当前 UI 只能查看 run 和指标，还没有“复制配置重新运行”或“从 checkpoint 恢复”的明确入口。
   5.2 目标
   Run 详情页提供“复制为新任务”入口。
   如果 Run 有 checkpoint，提供“从 checkpoint 恢复”入口。
   新 run 记录来源 run 和来源 checkpoint。
   不要求第一版自动保证数据、代码、镜像完全存在，只记录并提示缺失项。
   5.3 数据模型
   推荐在 ExperimentRun 增加：

ParentRunID        *uint `json:"parentRunID" gorm:"index"`
SourceCheckpointID *uint `json:"sourceCheckpointID" gorm:"index"`
ReproduceSnapshot  datatypes.JSONMap `json:"reproduceSnapshot" gorm:"type:jsonb"`
如果希望少改表结构，第一版可写入 Tags：

{
"parentRunID": 123,
"sourceCheckpointID": 456,
"reproduceMode": "copy-config"
}
5.4 后端 API
新增：

POST /api/v1/experiments/runs/:runID/reproduce
请求：

{
"mode": "copy-config",
"checkpointID": 456,
"name": "lr-1e-4-resume"
}
